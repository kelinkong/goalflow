# GoalFlow 1.0.1 人文关怀功能拆解文档

**项目名称**：GoalFlow  
**目标版本**：1.0.1  
**文档日期**：2026-03-31  
**文档类型**：版本需求拆解 + 实施规划

## 一、版本目标

1.0.1 不是继续增强“监督感”，而是让 GoalFlow 更像一个能陪人度过起伏的记录工具。

本版本只做 4 件事：

1. 低能量模式
2. 轻量复盘
3. 合法化休息 / 调整状态
4. 从记录走向理解

其中：

1. “低能量模式”和“休息 / 调整状态”属于同一套产品哲学，需要一起落地
2. 不做“今天只看一件事”
3. 不做复杂 AI 教练式建议
4. 优先做低负担、可上线的小而完整能力

## 二、产品原则

1. 允许用户状态有起伏
2. 允许“记录”轻于“推进”
3. 允许休息被看见，而不是被误判成失败
4. 产品输出以观察为主，不以指挥为主
5. 所有新能力都应降低心理负担，而不是增加行为压力

## 三、版本范围

### 3.1 本版本要做

1. 首页支持进入“低能量模式”
2. 当天支持标记为“调整日”
3. 复盘支持轻量记录
4. 轨迹页支持展示“调整日”
5. 系统按周输出简单观察文案
6. 文案和交互统一成“陪伴式”表达

### 3.2 本版本不做

1. AI 自动建议行动方案
2. 复杂情绪分析
3. 独立日记系统
4. 新增社交能力
5. 多层级能量模型

## 四、功能拆解

## 4.1 功能一：低能量模式

### 目标

当用户状态差、精力不足、被现实打断时，仍然愿意打开 GoalFlow，而不是因为“今天做不完”而逃开。

### 用户价值

1. 用户可以主动承认“今天状态有限”
2. 产品不再默认每一天都应该高完成度
3. 用户依然可以留下记录，而不是彻底断开

### 最小产品方案

首页新增一个当天状态入口：

1. 正常节奏
2. 低能量模式

当用户切换为低能量模式后：

1. 首页出现轻提示，如“今天先保留最小动作也可以”
2. 轨迹页当天以“调整日”展示
3. 不强推复盘深度和完成强度
4. 不影响目标、习惯、复盘本身的正常记录

### 交互建议

首页可放在日期卡片或复盘卡附近，采用轻按钮或 pill 形式，不要像“警报按钮”。

文案建议：

1. 今天状态一般
2. 切到低能量模式
3. 今天先保留最小动作

### 技术落点

建议新增按日状态记录，而不是只做本地 UI 状态。

建议新增实体：

`DailyState`

建议字段：

1. `id`
2. `user_id`
3. `date`
4. `energy_mode`，取值建议：
   - `NORMAL`
   - `LOW_ENERGY`
5. `day_type`，取值建议：
   - `NORMAL`
   - `ADJUSTMENT`
6. `note`，可空，预留
7. `created_at`
8. `updated_at`

说明：

1. `energy_mode` 表示当天主观状态
2. `day_type` 表示当天是否主动标记为调整日
3. 两者可以先放在同一张表，避免过早拆分

## 4.2 功能二：轻量复盘

### 目标

让用户在不想深度书写时，仍然能留下足够轻的记录。

### 用户价值

1. 降低复盘门槛
2. 让“留下痕迹”比“写完整”更优先
3. 增强用户回来的意愿

### 最小产品方案

复盘页增加轻量模式，至少支持以下三种之一：

1. 选一个今天的状态 / 情绪
2. 写一句话
3. 记录一个标签

轻量复盘和完整复盘共享同一入口，不要拆成两个独立系统。

### 交互建议

复盘页顶部增加轻量入口：

1. 今天的感受
2. 一句话记录
3. 今天想带走的一个词

完整复盘继续保留，但允许用户只提交轻量内容。

### 数据设计建议

当前 `daily_reviews` / `daily_review_items` 已存在，本版本不建议新建第二套复盘表。

建议在 `daily_reviews` 上补充轻量字段：

1. `mood_tag VARCHAR(32) NULL`
2. `short_note VARCHAR(300) NULL`
3. `review_mode VARCHAR(16) NOT NULL DEFAULT 'FULL'`

取值建议：

1. `FULL`
2. `LIGHT`

说明：

1. 当用户只填轻量内容时，`review_mode = LIGHT`
2. 当用户填写完整四维复盘时，`review_mode = FULL`
3. 同一天仍保持一条复盘主记录

### 后端改动

1. `DailyReview` 实体补字段
2. DTO / Request 补字段
3. `schema.sql` 补列
4. 保存逻辑允许轻量字段单独提交

### 前端改动

1. `daily_review_screen.dart` 增加轻量模式 UI
2. 首页“今日复盘”卡支持显示“今天已留下一点记录”
3. 轨迹详情页支持展示轻量复盘内容

## 4.3 功能三：合法化休息 / 调整状态

### 目标

把“今天没有推进”从失败中拆出来，承认真实生活里存在休息、缓冲和被打断。

### 用户价值

1. 日历不再只区分完成 / 未完成
2. 轨迹更真实
3. 用户不会因为某一天状态特殊而产生羞耻感

### 最小产品方案

当用户主动标记当天为“调整日”后：

1. 日历格子有独立样式
2. 轨迹页提示把这类天视为“节奏调整”
3. 日历详情页显示“这一天被标记为调整日”

### 展示建议

日历层面建议至少有三种状态：

1. 有记录 / 有推进
2. 调整日
3. 无记录

注意：

1. 调整日不能伪装成完成
2. 调整日也不能在视觉上等同于“空白失败”

### 技术建议

复用 `DailyState.day_type` 即可，不再额外新增单独表。

前端聚合时：

1. 先看当天是否为 `ADJUSTMENT`
2. 再看目标 / 习惯 / 复盘数据
3. 日历展示规则优先级要明确

建议优先级：

1. 调整日状态优先于热力图背景
2. 原有目标/习惯/复盘点仍可保留
3. 如果当天既是调整日又有记录，应显示“调整中仍有记录”

## 4.4 功能四：从记录走向理解

### 目标

让 GoalFlow 不只记录行为，还能用非常克制的方式帮助用户看见自己的节奏。

### 用户价值

1. 用户获得被理解的感受
2. 数据不只是堆积，而是开始有意义
3. 产品价值从“记账工具”走向“理解工具”

### 最小产品方案

先不引入 AI，采用规则型观察。

每周生成 1 到 3 条观察，不给命令式建议，只给事实观察。

示例：

1. 最近一周，习惯完成比目标推进更稳定。
2. 你更容易在周中留下记录。
3. 有复盘的那几天，总分通常更高。

### 展示位置建议

优先放在轨迹页趋势卡附近，作为“本周观察”模块。

不要放在首页主信息区，避免用户一打开就被分析。

### 技术建议

首版直接前端规则生成即可，基于已有数据聚合：

1. 目标完成记录
2. 习惯完成记录
3. 复盘记录
4. 调整日记录

规则建议：

1. 周内各类记录天数统计
2. 周内每日总分对比
3. 工作日 / 周末差异
4. 复盘日与非复盘日分数差异

等规则稳定后，再考虑后端汇总接口。

## 五、页面改动拆解

## 5.1 首页

新增：

1. 今日状态入口
2. 低能量模式提示
3. 复盘卡支持轻量记录状态文案

调整：

1. 文案弱化压力感
2. 当天为调整日时，首页不要再用“待完成”语气强调任务

## 5.2 复盘页

新增：

1. 轻量模式入口
2. 情绪 / 一句话 / 标签
3. 轻量保存逻辑

调整：

1. 校验规则放宽
2. 轻量模式下不强制完整四维输入

## 5.3 轨迹页

新增：

1. 调整日展示
2. 本周观察卡片
3. 轻量复盘显示适配

调整：

1. 日历格子状态优先级
2. 轨迹提示文案继续保持中性、陪伴式

## 5.4 设置页

可选新增：

1. 是否展示本周观察
2. 提醒文案风格开关

说明：

这两项不属于 1.0.1 必须项，可以放在后续版本。

## 六、后端与数据改动清单

1. `daily_reviews` 补列：
   - `mood_tag`
   - `short_note`
   - `review_mode`
2. 新增 `daily_states` 表
3. 新增对应实体、DTO、Mapper、Service、Controller
4. `schema.sql` 同步更新
5. 线上数据库补列 / 建表
6. 相关导出逻辑同步支持新字段

## 七、接口建议

## 7.1 Daily State 接口

建议新增：

1. `GET /api/daily-states/{date}`
2. `PUT /api/daily-states/{date}`
3. `GET /api/daily-states/calendar?month=YYYY-MM`

返回内容建议：

1. `date`
2. `energyMode`
3. `dayType`
4. `note`

## 7.2 Daily Review 接口

现有接口保持路径不变，但请求结构补充：

1. `moodTag`
2. `shortNote`
3. `reviewMode`

## 八、前端状态管理改动

`AppState` 需要新增：

1. `Map<String, DailyState> _dailyStatesByDate`
2. `Map<String, Set<String>> _adjustmentDatesByMonth`
3. `bool isDailyStateMonthLoaded(DateTime month)`
4. `DailyState? getDailyState(DateTime date)`
5. `Future<void> fetchDailyState(DateTime date)`
6. `Future<void> fetchDailyStateCalendar(DateTime month)`
7. `Future<void> saveDailyState(...)`

说明：

轨迹页月历、首页状态入口、本周观察都会依赖这部分状态。

## 九、开发顺序建议

### 第一阶段：底层数据打通

1. 后端新增 `daily_states`
2. `daily_reviews` 补字段
3. `schema.sql` 更新
4. 前端 `AppState` 接入新接口

### 第二阶段：低能量模式 + 调整日

1. 首页状态切换
2. 轨迹页日历展示调整日
3. 当天详情支持查看调整状态

### 第三阶段：轻量复盘

1. 复盘页增加轻量模式
2. 首页复盘卡适配
3. 轨迹详情适配

### 第四阶段：本周观察

1. 前端聚合近 7 天数据
2. 生成 1 到 3 条规则型观察
3. 展示在轨迹页

## 十、验收标准

## 10.1 低能量模式

1. 用户可以在首页为当天切换低能量模式
2. 该状态刷新后不丢失
3. 轨迹页能正确显示调整日

## 10.2 轻量复盘

1. 用户只填轻量内容也能保存
2. 首页和轨迹页能正确展示轻量复盘
3. 不会破坏原完整复盘流程

## 10.3 调整日

1. 调整日能与无记录区分开
2. 调整日不会被错误算成完成
3. 日历和详情页展示一致

## 10.4 本周观察

1. 至少能稳定输出 1 条观察
2. 观察内容基于真实数据
3. 观察语气不带命令和评判

## 十一、文档同步要求

本版本开发时必须同步更新以下文档与文件：

1. 本文档
2. `docs/content-tone-guide.md`
3. 后端 `schema.sql`
4. 如接口变化明显，补充或更新技术设计文档

## 十二、版本结论

1.0.1 的重点不是增加更多功能，而是修正产品对人的基本态度。

它要让 GoalFlow 从“记录你有没有做到”往前走一步，变成“记录你怎样走过这一天，并帮你看见自己的节奏”。
